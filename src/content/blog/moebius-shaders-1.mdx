## Pipeline

This project uses NPR (Non-Photorealistic Rendering).
Instead of simulating physical light, the goal is graphic legibility, with
flat colors, controlled shadows, and ink-like outlines.
All object shaders are unlit: no light probes, no global illumination.
Each one computes its own shading from a single directional light,
with lit and shadow colors defined directly by the artist.

The pipeline has two layers.
The first is a set of object shaders (color gradient, hatching, bubble, skydome), all unlit.
The second is a fullscreen post-process pass that runs after the scene is rendered
and reads the depth, normal, and color buffers to draw outlines on top.

## Outline

Outlines are computed as a fullscreen post-process pass.
The pass runs after the scene is fully rendered
and detects edges by reading the depth, normal, and color buffers.
No mesh modification needed: the detector works on pixels, not geometry.

The starting point was a [tutorial by Digvijaysinh Gohil](https://www.youtube.com/watch?v=nc3a3THBFrg)
covering two detection methods: depth-based and normal-based.
Both produced clean results on solid geometry.

### Depth-based detection

The depth buffer stores, for each pixel, how far the camera is from the nearest surface.
At a silhouette or a hard geometric crease, this value jumps sharply between neighboring pixels.
A gradient filter makes those discontinuities visible.

The filter used is the Scharr operator, a 3×3 convolution applied to the depth buffer.
Scharr is a variant of the [Sobel operator](https://en.wikipedia.org/wiki/Sobel_operator):
it uses coefficients 3 and 10 instead of Sobel's 1 and 2,
which produces a more isotropic gradient, responding more uniformly to edges at all orientations.

$$
G_x = \begin{bmatrix} -3 & 0 & +3 \\ -10 & 0 & +10 \\ -3 & 0 & +3 \end{bmatrix} * I
\qquad
G_y = \begin{bmatrix} -3 & -10 & -3 \\ 0 & 0 & 0 \\ +3 & +10 & +3 \end{bmatrix} * I
$$

$$|G| = \sqrt{G_x^2 + G_y^2}$$

A pixel becomes an edge if $|G|$ exceeds a threshold.

The code below visualizes this directly as grayscale, brighter means closer.
In Unity this buffer already exists, no raymarching is needed.

```glsl live
precision mediump float;
uniform vec2 u_resolution;

float sdSphere(vec3 p, float r) { return length(p) - r; }

// Three spheres at clearly different distances from the camera
float scene(vec3 p) {
    float close = sdSphere(p - vec3(-0.5,  0.1,  1.0), 0.45);
    float middle = sdSphere(p - vec3( 0.4, -0.2,  0.0), 0.35);
    float far = sdSphere(p - vec3(-0.1,  0.4, -0.9), 0.3);
    return min(close, min(middle, far));
}

// Returns ray travel distance, or 8.0 on miss
float getDepth(vec2 uv) {
    vec3 ro = vec3(0.0, 0.0, 3.0);
    vec3 rd = normalize(vec3(uv, -1.5));
    float t = 0.0;
    for (int i = 0; i < 48; i++) {
        float d = scene(ro + rd * t);
        if (d < 0.001) return t;
        t += d;
        if (t > 8.0) break;
    }
    return 8.0;
}

void main() {
    vec2 uv = (gl_FragCoord.xy - 0.5 * u_resolution) / u_resolution.y;
    float depth = getDepth(uv);

    // Remap to brightness: close = bright, far = dark, miss = near black
    float normalized = 1.0 - clamp((depth - 1.5) / 3.0, 0.0, 1.0);
    vec3 col = depth < 8.0 ? vec3(normalized) : vec3(0.04);

    gl_FragColor = vec4(col, 1.0);
}
```

The second one applies the Scharr filter to those depth values.
It samples the 8 neighboring pixels, computes the Scharr gradient magnitude from those depth values,
and marks a pixel as an edge if that magnitude exceeds a threshold.

```glsl live
precision mediump float;
uniform vec2 u_resolution;

float sdSphere(vec3 p, float r) { return length(p) - r; }

float scene(vec3 p) {
    float close = sdSphere(p - vec3(-0.5,  0.1,  1.0), 0.45);
    float middle = sdSphere(p - vec3( 0.4, -0.2,  0.0), 0.35);
    float far = sdSphere(p - vec3(-0.1,  0.4, -0.9), 0.3);
    return min(close, min(middle, far));
}

float getDepth(vec2 uv) {
    vec3 ro = vec3(0.0, 0.0, 3.0);
    vec3 rd = normalize(vec3(uv, -1.5));
    float t = 0.0;
    for (int i = 0; i < 48; i++) {
        float d = scene(ro + rd * t);
        if (d < 0.001) return t;
        t += d;
        if (t > 8.0) break;
    }
    return 8.0;
}

void main() {
    vec2 uv = (gl_FragCoord.xy - 0.5 * u_resolution) / u_resolution.y;
    float px = 1.0 / u_resolution.y;

    float dCenter = getDepth(uv);

    // Sample the 8 neighbors
    float d00 = getDepth(uv + vec2(-px, -px));
    float d10 = getDepth(uv + vec2(  0, -px));
    float d20 = getDepth(uv + vec2( px, -px));
    float d01 = getDepth(uv + vec2(-px,   0));
    float d21 = getDepth(uv + vec2( px,   0));
    float d02 = getDepth(uv + vec2(-px,  px));
    float d12 = getDepth(uv + vec2(  0,  px));
    float d22 = getDepth(uv + vec2( px,  px));

    // Scharr gradient
    float gx = -3.0*d00 + 3.0*d20 - 10.0*d01 + 10.0*d21 - 3.0*d02 + 3.0*d22;
    float gy = -3.0*d00 - 10.0*d10 - 3.0*d20 + 3.0*d02 + 10.0*d12 + 3.0*d22;
    float edge = step(5.0, sqrt(gx*gx + gy*gy));

    float fill = dCenter < 8.0 ? 0.15 : 0.04;
    vec3 col = vec3(mix(fill, 1.0, edge));

    gl_FragColor = vec4(col, 1.0);
}
```

Here is the same logic in the actual Unity HLSL.
The Scharr kernels are shared across all three detection methods, so they live in a struct:

```hlsl
struct ScharrOperators
{
    float3x3 x;
    float3x3 y;
};

ScharrOperators GetEdgeDetectionKernels()
{
    ScharrOperators kernels;

    kernels.x = float3x3(
        -3, -10, -3,
         0,   0,  0,
         3,  10,  3
    );

    kernels.y = float3x3(
        -3,  0,  3,
       -10,  0, 10,
        -3,  0,  3
    );

    return kernels;
}

void DepthBasedOutlines_float(float2 screenUV, float2 px, out float outlines)
{
    outlines = 0.0;

    ScharrOperators k = GetEdgeDetectionKernels();

    float gx = 0.0;
    float gy = 0.0;

    for (int i = -1; i <= 1; i++)
    {
        for (int j = -1; j <= 1; j++)
        {
            float2 offset = float2(i, j) * px;

            float d = SampleSceneDepth(screenUV + offset);
            d = LinearEyeDepth(d, _ZBufferParams);

            // (row, col) -> (j+1, i+1)
            gx += d * k.x[j + 1][i + 1];
            gy += d * k.y[j + 1][i + 1];
        }
    }

    outlines = step(0.05, sqrt(gx * gx + gy * gy));
}
```

### Normal-based detection

The normal buffer stores, for each pixel, a unit vector pointing perpendicular to the surface.
On a flat area, neighboring pixels have nearly identical normals: their dot product is close to 1.
At a crease or fold, the normals suddenly diverge, and the dot product drops sharply.
Applying the Scharr filter to these dot product values detects exactly those orientation discontinuities.

{/* ILLUSTRATION: schéma - deux surfaces qui se rejoignent en angle, vecteurs normaux dessinés sur chaque face, dot product ~1 sur la surface plane, dot product bas à la crête */}

```hlsl
void NormalBasedOutlines_float(float2 screenUV, float2 px, out float outlines)
{
    outlines = 0.0;

    ScharrOperators k = GetEdgeDetectionKernels();

    float gx = 0.0;
    float gy = 0.0;

    float3 centerNormal = SampleSceneNormals(screenUV);

    for (int i = -1; i <= 1; i++)
    {
        for (int j = -1; j <= 1; j++)
        {
            if (i == 0 && j == 0) continue;

            float2 offset = float2(i, j) * px;
            float3 n = SampleSceneNormals(screenUV + offset);

            // Dot product measures angular difference from center normal
            float dp = dot(centerNormal, n);

            gx += dp * k.x[j + 1][i + 1];
            gy += dp * k.y[j + 1][i + 1];
        }
    }

    // Lower threshold: dp values are close to 1 everywhere except at creases
    outlines = step(0.2, sqrt(gx * gx + gy * gy));
}
```

This catches detail that depth-based detection misses:
interior creases, folds, and surface ridges where depth barely changes but the orientation shifts sharply.

Both methods work on solid geometry. But what happens when a transparent object is in the scene?

{/* IMAGE: capture d'un objet transparent dans la scène, sans outline - légende : "where did the outline go?" */}

As you can see, transparent geometry doesn't write to the depth or normal buffers,
so both filters see nothing where the object should be.
This means that glass, bubbles, or any surface that lets light through will simply never get outlines.

### Color-based detection

The fix is to read from the color buffer instead.
Transparent objects do render to color, even if they skip depth and normals.
This third method samples four neighbors (left, right, up, down)
and measures luminance contrast across them.
A `smoothstep` replaces the hard `step`, which softens the outline boundary and reduces aliasing.

```hlsl
float SG_Luma(float3 c)
{
    // Standard perceptual luminance weights
    return dot(c, float3(0.2126, 0.7152, 0.0722));
}

void ColorBasedOutlinesFromSamples_float(
    float4 cL, float4 cR, float4 cU, float4 cD,
    float threshold,
    out float outlines)
{
    float lL = SG_Luma(cL.rgb);
    float lR = SG_Luma(cR.rgb);
    float lU = SG_Luma(cU.rgb);
    float lD = SG_Luma(cD.rgb);

    float lMin = min(min(lL, lR), min(lU, lD));
    float lMax = max(max(lL, lR), max(lU, lD));

    // Local luminance contrast, amplified before thresholding
    float e = (lMax - lMin) * 5.0;

    outlines = smoothstep(threshold, threshold * 2.0, e);
}
```

### Resolution awareness

The outline pass samples neighbors at a UV offset of `OutlinesThickness / screen_resolution`.
This matters because UV space is resolution-independent:
the same offset covers twice as many physical pixels at 4K as at 1080p.
Dividing by resolution ensures a thickness of 1.0 always means exactly one pixel on screen,
whatever the rendering resolution.

{/* ILLUSTRATION: schéma - même offset UV sur deux résolutions différentes : à 1080p il couvre 1px, à 4K il couvre 2px sans la correction; avec la correction, les deux donnent 1px */}

### Limitations

Depth-based detection aliases on thin geometry
and can miss surfaces seen nearly edge-on, where depth barely changes across the silhouette.
It also fails when two surfaces are at nearly the same depth:
the gradient sees no discontinuity, even if the geometry is distinct.
Normal-based detection doesn't have that last problem,
but it can fire on smooth curved surfaces where normals shift gradually,
producing faint outlines where none were intended.

{/* IMAGE: thin geometry (wire, branches) with broken/dotted outline - depth aliasing */}
{/* IMAGE: grazing angle surface - outline disappears at shallow viewing angle */}
{/* IMAGE: two objects at similar depth - no outline between them */}

The inverted hull technique avoids most of these problems:
each mesh is rendered twice, once normally, then scaled up slightly with backfaces visible,
producing a solid halo around the geometry.
It's stable and gives clean silhouettes even on thin geometry and at shallow angles.
The tradeoff is that it only catches outer contours:
interior creases, folds, and surface details are invisible to it.
A hybrid approach (hull for silhouettes, screen-space for interior detail) could combine the best of both.

## Hatching

The hatching shader is closely based on the technique shown in
[this video by Useless Game Dev](https://www.youtube.com/watch?v=jlKNOirh66E)
(which by the way was also the initial spark for the project as a whole).
The only addition in my version is a secondary outline drawn around cast shadows.

### Hatch texture

All hatching patterns are packed into a single texture.
Each RGB channel holds a different layer: one angle for the lightest shadow zone,
a denser or perpendicular pattern for the next zone, and so on.
Sampling three independent channels from one texture costs the same as sampling one.

{/* IMAGE: hatch texture visualized per channel - R, G, B shown separately, then combined */}

### Brightness zones

The shader reads pixel brightness and posterizes it into discrete zones.
Each zone maps to a combination of texture channels.

{/* ILLUSTRATION: brightness gradient -> posterized zones -> corresponding hatch layers (similar to the video diagram) */}

Lit pixels show the base color.
As brightness drops, the shader blends in one channel, then two,
adding lines and cross-hatching progressively.

### Limitation

The texture is sampled in screen space.
Useless Game Dev notes it himself: in an ideal world, the crosshatch pattern would follow the curve of the object it's on.
Here it doesn't: move the camera, and the lines slide across the surface rather than staying anchored to it.
On a still frame this is invisible, but it becomes noticeable on a moving camera.

One approach is to embed the hatching lines into the normal buffer so they read as surface detail.
The problem is that the Scharr filter can't distinguish a geometric crease from a hatch line:
both produce edges, and the mesh ends up covered in unintended outlines.
Resolving this would require a separate buffer or a masking pass (maybe a project for later?).

The playground below illustrates the difference.
Both halves use the same orbiting camera driven by `u_time`.
The top sphere computes the hatch pattern from screen coordinates:
the lines slide across the surface as the camera moves.
The bottom sphere uses the world-space position of the hit point:
the lines stay anchored to the sphere regardless of camera angle.

```glsl live tall
precision mediump float;
uniform vec2 u_resolution;
uniform float u_time;

float sdSphere(vec3 p, float r) { return length(p) - r; }

vec3 calcNormal(vec3 p) {
    vec2 e = vec2(0.001, 0.0);
    return normalize(vec3(
        sdSphere(p + e.xyy, 1.0) - sdSphere(p - e.xyy, 1.0),
        sdSphere(p + e.yxy, 1.0) - sdSphere(p - e.yxy, 1.0),
        sdSphere(p + e.yyx, 1.0) - sdSphere(p - e.yyx, 1.0)
    ));
}

void main() {
    // Split the canvas into two halves, one sphere per half
    float halfH = u_resolution.y * 0.5;
    bool isTop = gl_FragCoord.y >= halfH;
    float PI = 3.14159265;

    // Re-center UV within each half so both spheres appear the same size
    float localY = isTop ? gl_FragCoord.y - halfH : gl_FragCoord.y;
    vec2 uv = (vec2(gl_FragCoord.x, localY) - 0.5 * vec2(u_resolution.x, halfH)) / halfH;

    // Orbit the camera around the sphere using u_time
    float angle = u_time * 0.5;
    vec3 ro = vec3(sin(angle) * 2.5, 0.6, cos(angle) * 2.5);
    vec3 forward = normalize(-ro);
    vec3 right = normalize(cross(vec3(0.0, 1.0, 0.0), forward));
    vec3 up = cross(forward, right);
    vec3 rd = normalize(forward + uv.x * right + uv.y * up);

    // Sphere raymarching
    float t = 0.0;
    bool hit = false;
    for (int i = 0; i < 64; i++) {
        float d = sdSphere(ro + rd * t, 1.0);
        if (d < 0.001) { hit = true; break; }
        t += d;
        if (t > 10.0) break;
    }

    vec3 col = vec3(0.06);
    if (hit) {
        vec3 p = ro + rd * t;
        vec3 n = calcNormal(p);
        vec3 light = normalize(vec3(1.0, 1.5, 1.0));
        float NdotL = dot(n, light);

        // lw is line width: the dark stripe occupies the first lw fraction of each period
        float lw = 0.15;
        float h1 = 1.0;
        float h2 = 1.0;
        float h3 = 1.0;

        if (isTop) {
            // Top: pattern from screen coordinates, slides with the camera
            float scale = 12.0;
            h1 = step(lw, fract((gl_FragCoord.x + gl_FragCoord.y) / scale));
            h2 = step(lw, fract((gl_FragCoord.x - gl_FragCoord.y) / scale));
            h3 = step(lw, fract((gl_FragCoord.x * 2.0 + gl_FragCoord.y) / scale));
        } else {
            // Bottom: pattern from spherical coordinates, anchored to the surface
            float theta = atan(p.z, p.x) / PI; // longitude [-1, 1]
            float phi = acos(clamp(p.y, -1.0, 1.0)) / PI; // colatitude [0, 1]
            float scale = 22.0;
            h1 = step(lw, fract((theta + phi) * scale));
            h2 = step(lw, fract((theta - phi) * scale));
            h3 = step(lw, fract((theta * 2.0 + phi) * scale));
        }

        // Add hatch layers progressively as the surface gets darker
        float hatch = 0.0;
        if (NdotL < 0.65) hatch = max(hatch, 1.0 - h1);
        if (NdotL < 0.30) hatch = max(hatch, 1.0 - h2);
        if (NdotL < 0.05) hatch = max(hatch, 1.0 - h3);

        col = mix(vec3(0.95), vec3(0.05), hatch);
    }

    gl_FragColor = vec4(col, 1.0);
}
```

The outline and hatching passes are the core of the effect. Everything else (bubble shader, color gradient, skydome) builds on top of the same pipeline and is covered in part 2.
